<objective>
Close the D-03 re_entry_stage allowlist gap at a single enforcement layer, then point the MCP `author` tool's amend/supersede handlers at the already-correct `LifecycleService` (D-01), reconcile the tool params (`re_entry_stage`, `new_slug`), require `reason` on amend (D-04), emit a next-step hint (D-05), and rewrite the tool teaching so an MCP-only agent lands the spec correctly (D-03 / D-07 MCP half). The MCP reroute is a call-site swap — `client.Lifecycle` already exists on the MCP `Client`.

The re-entry allowlist fix and the MCP reroute are deliberately co-located in this one plan so the MCP exposure vector and its validation land atomically: an MCP `author action=amend` can never ship without the handler rejecting invalid re-entry stages. Enforcement uses **Option B** (reviewer-preferred): a shared `SpecStage.IsValidReEntryStage()` helper called from BOTH the `TransitionAmend` handler (the single RPC gate that CLI and MCP both traverse) and storage `LifecycleAmendSpec` (defense-in-depth via the same helper, so the two cannot diverge). This resolves the review HIGH finding — the prior `IsValid()`+`ExcludesReEntry()` check wrongly accepted `approved`/`in_progress`/`review`.

Purpose: Bring the MCP surface — the canonical authoring path for MCP-only agents — onto the single correct lifecycle implementation with a single-source-of-truth re-entry allowlist so the inverted semantics and the #899 no-op cannot recur.
Output: `IsValidReEntryStage` helper + handler/storage enforcement; rerouted `handleAmend`/`handleSupersede`, renamed and re-documented params, next-step hint, `authoringStageFromString` deleted, tool tests rewritten against a lifecycle mock.

Depends on 07-02: the MCP reroute must not ship before the claim-release lands (D-08), otherwise an MCP-driven in-flight amend would leave a lingering claim/lease (review ordering finding). Depends on 07-01 for the `TransitionSupersedeRequest.reason` proto field the rerouted supersede path threads.
</objective>
