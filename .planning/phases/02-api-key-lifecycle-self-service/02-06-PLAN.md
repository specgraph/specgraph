<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator CLI → ResyncUserRole | Admin-gated (user.manage) write of another user's role |
| DB role write → standing keys | Live-role read at resolveAPIKey propagates the demotion |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-20 | Elevation of Privilege | Resync used to escalate rather than demote | high | mitigate | Mapped to user.manage (admin-only, Plan 04); role is an explicit operator input, re-derived downward; no self/owner path |
| T-02-21 | Spoofing / Access Control | Non-admin invoking resync | high | mitigate | Cedar user.manage gate rejects non-admin principals before the handler runs |
| T-02-22 | Repudiation | No trace of a forced demotion/off-board | medium | mitigate | Structured audit line on resync with operator as actor (per RESEARCH §6 audit guidance); never log token material |
| T-02-23 | Elevation (latency) | Demotion "un-sticks" on next login (convergence trade-off) | low | accept | D-03 documented convergence; `--revoke-keys` is the hard off-board for true termination |
</threat_model>
