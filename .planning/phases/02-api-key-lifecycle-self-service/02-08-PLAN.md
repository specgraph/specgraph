<objective>
Ship the net-new "MCP Keys" SvelteKit dashboard panel (D-07 web surface): an `identityClient` wired into `web/src/lib/api/client.ts` with a CSRF double-submit interceptor, a `keys.svelte.ts` state module, a `/keys` route with list/create/rotate/revoke, and a one-time reveal modal for the minted secret. All mutations authenticate via the `specgraph_session` cookie, which the Connect interceptor reads directly on the server (`internal/auth/interceptor.go:57-81` — NOT the REST `cookieToAuthHeader` whoami path, cursor #2), and carry the CSRF token per D-09 (the `specgraph_csrf` cookie is issued on the whoami GET the app calls on load, Plan 03; the token is validated by the `csrfValidate` mount on the Connect handler, Plan 05).

Purpose: Deliver SC#1's web half — a user self-provisions keys from the dashboard (currently read-only) — with the one-time-reveal and CSRF safety the design mandates.
Output: identityClient + CSRF interceptor, keys state module, /keys route, RevealKeyModal, Vitest tests. Ends with a human-verify checkpoint for the interactive UI + one-time reveal.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → cookie-authed mutation | POST self-mint/rotate/revoke from the dashboard needs CSRF defense |
| server response → browser | One-time plaintext must not be persisted or re-fetchable |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-02-27 | Tampering (CSRF) | Cookie-auth'd web mutations | high | mitigate | Double-submit CSRF interceptor echoes specgraph_csrf cookie into X-CSRF-Token; Plan 03 middleware validates; verified in checkpoint |
| T-02-28 | Information Disclosure | Minted secret persisted in DOM/storage | medium | mitigate | Reveal modal shows plaintext once, clears on close; state module never stores/re-fetches it |
| T-02-29 | Elevation of Privilege | Web path bypassing server-side floor/source gates | high | mitigate | Panel consumes the same four self RPCs whose handler (Plan 05) enforces owner-from-context, source gate, and RoleMin floor; no client-trusted authorization |
| T-02-30 | Spoofing | Session cookie theft via XSS reading CSRF cookie | low | accept | CSRF cookie is intentionally non-HttpOnly for double-submit; session cookie stays HttpOnly; standard XSS hygiene applies |
</threat_model>
