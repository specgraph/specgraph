<objective>
Deliver the AUTH-04 token-validation half: RFC 8707 audience/resource-bound validation for MCP
JWTs (D-05.3) and an RFC 7662 introspection path for opaque tokens (D-06) ŌĆö both feeding the Plan
01 shared `materializeIdentity`, while keeping the static-credential prefix dispatch fully additive
(D-08).

Scope:
1. Audience reconciliation (Pitfall 2 / OQ2): keep the existing per-issuer verifier's signature +
   issuer + exp validation (aud == client_id for web login), then add an explicit post-verify
   assertion that, for the MCP resource-server path ONLY (request marked auth.WithMCPRequest by
   Plan 03's /mcp/ wrapper), the token's `aud` contains the canonical MCP resource URI. A token bound
   only to `client_id` and not the resource URI is rejected. Non-MCP ConnectRPC JWT callers are never
   subjected to this check even when MCP RS is enabled (review HIGH #2).
2. Introspection branch (Pattern 5): add a branch in `Resolve` so an opaque bearer that is neither
   `spgr_sk_`/`spgr_ws_` nor JWT-shaped attempts introspection against introspection-capable
   providers, checks active + audience, then materializes identity via the shared helper. Ordering
   adds an EXPLICIT `strings.HasPrefix(token, apiKeyPrefix)` guard routing `spgr_sk_` to
   `resolveAPIKey` BEFORE introspection (today it only reaches resolveAPIKey by fallthrough) so API
   keys never hit the IdP (review HIGH #3, D-08); introspection slots in after that guard, gated on
   config presence.
3. Thread the SINGLE hoisted canonical resource URI + per-provider introspection config into
   `NewIdentityStore` in serve.go ŌĆö reusing Plan 03's hoisted mcpResourceURI/mcpRSEnabled, not a
   second computation.

Purpose: The delegated-auth token validation that makes SpecGraph a spec-correct OAuth 2.1
resource server without minting its own tokens or breaking existing API keys/sessions.
Output: Audience + introspection validation with unit coverage, green under `task test`.
</objective>
