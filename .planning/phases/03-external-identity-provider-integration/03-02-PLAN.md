<objective>
Deliver AUTH-01: a native, generic OAuth2 + userinfo login provider (GitHub as the driving
config), plugging into the Plan 01 seam so it produces `*OIDCClaims` that flow through the exact
same binding/JIT/claims-mapping machinery OIDC uses.

Scope:
1. Relax `BuildLoginProviders` so it constructs an `oauth2LoginProvider` for `kind=="oauth2"`
   instead of hard-rejecting it (D-01). GitHub ships as documented config, not a hardcoded preset.
2. New `oauth2LoginProvider` (Pattern 1): reuses `oauth2.Config` + PKCE/state for the authorize
   URL and code exchange, but replaces id_token verification with a userinfo GET and (when the
   primary email is private) a primary&&verified email fetch (D-02). Subject = the configured
   stable id selector; Issuer = a synthetic provider id (D-09). Roles via claims_mapping over the
   userinfo body (D-04).

Purpose: A second, non-OIDC IdP path so users log in via GitHub-direct with no broker, using the
same session model — and the data layer stores bindings the same `(issuer, subject) → user_id`
way, keeping N-providers-per-user possible (D-03) without building the linking flow (deferred).
Output: The oauth2 provider + unit coverage, green under `task test`.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| SpecGraph → GitHub token/userinfo/emails endpoints | Server-side outbound calls carrying the client secret + access token |
| userinfo response → identity binding | Attacker-influenced profile fields cross into subject/email/role derivation |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-02-01 | Spoofing | subject selection | high | mitigate | Bind on the stable id selector (SubjectField, e.g. numeric id), never a renameable username (D-02); asserted by the oauth2 provider test. |
| T-03-02-02 | Spoofing | email trust | high | mitigate | Only a primary&&verified email is accepted; unverified/absent → blank Email (D-02); asserted by the verified-email fallback test. |
| T-03-02-03 | Tampering | authorization-code interception | high | mitigate | PKCE S256 + state reused from the existing login machinery; no nonce needed (no id_token). |
| T-03-02-04 | Information Disclosure | SSRF via userinfo/emails URLs | medium | mitigate | Endpoint URLs are operator-supplied config, validated non-empty at startup, never derived from token/user input; bounded-timeout client. |
| T-03-02-05 | Elevation of Privilege | cross-provider auto-link | high | mitigate | Do NOT auto-link across providers this phase; bindings stay per (issuer, subject) (D-03); linking flow deferred. |
| T-03-02-06 | Elevation of Privilege | issuer split → silent role-mapping failure (review HIGH #1) | high | mitigate | claims.Issuer, the (issuer,subject) binding, and the claims-mapping key all derive from the ONE config.ProviderIssuer(pc) helper, so a GitHub claims_mapping role reliably resolves instead of silently falling through to the default role; asserted by the ProviderIssuer-keyed ClaimsMapping test. |
| T-03-02-SC | Tampering | package installs | high | accept | No new packages (GitHub endpoints are config URLs, not registry artifacts — RESEARCH Package Legitimacy Audit: none). |
</threat_model>
