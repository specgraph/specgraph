<objective>
Deliver the AUTH-04 OAuth 2.1 resource-server protocol surface: the RFC 9728 Protected Resource
Metadata endpoint and the WWW-Authenticate challenge on unauthenticated MCP requests, so standard
MCP clients auto-discover the authorization server(s) (MCP Authorization spec rev 2025-06-18,
D-05.1/D-05.2/D-07).

Scope (Pattern 3):
1. `GET /.well-known/oauth-protected-resource` → 200 application/json with `resource` = the
   canonical MCP resource URI and `authorization_servers` = `config.ProviderIssuer(pc)` for every
   configured interactive/OAuth IdP (advertise all, D-07; the canonical helper reconciles the D-09
   synthetic oauth2 issuer — review HIGH #1). Public, unauthenticated.
2. A `/mcp/`-only auth wrapper that emits `WWW-Authenticate: Bearer resource_metadata="…"` on a
   401, replacing the bare 401 that `auth.RequireAuth` writes — scoped so `/api/*` and ConnectRPC
   401s are untouched (D-08, Pitfall 4). The wrapper also marks the request context with
   `auth.WithMCPRequest` before authenticating, so Plan 04's resource-URI audience check fires on the
   /mcp/ boundary only (review HIGH #2).
3. Canonical resource URI: derive once (above `NewIdentityStore` at serve.go:162) from
   `cfg.Auth.OIDC.MCPResourceURI`, defaulting to `<base_url>/mcp`. Enablement policy: if the derived
   URI is https, validate no-fragment and mount the RS surface; if it is non-https (e.g. the local
   http loopback default) and no explicit `mcp_resource_uri` was set, MCP OAuth RS stays DISABLED
   (metadata not mounted, /mcp/ keeps the bare 401) with a warn log — local dev is not blocked at
   boot. An explicit `mcp_resource_uri` MUST be https+no-fragment or boot aborts (startup-fatal,
   RESEARCH OQ1/A2; review MEDIUM dev/prod URI policy).
4. Reconcile the startup claims-mapping: re-key `buildClaimsMappingByIssuer` (serve.go) by
   `config.ProviderIssuer(pc)` so an oauth2 provider's mapping key equals its runtime claims.Issuer
   (review HIGH #1, D-04). This one-line serve.go change lives here (not Plan 02) because Plan 02 and
   Plan 03 share Wave 2 and only Plan 03 owns serve.go.

Purpose: The discovery handshake an MCP client needs before it can obtain and present a
resource-bound token; the token-validation half lands in Plan 04.
Output: The metadata handler + challenge wrapper mounted in serve.go, green under `task test`.

**D-05.4 error-model scope (W1 clarification):** D-05 item 4 locks a 401/403/400 error model. This
phase implements **401** (invalid/absent token → `WWW-Authenticate` challenge here; audience/introspection
failures → `ErrUnauthenticated` in Plan 04). **403 insufficient-scope is N/A** — SpecGraph authorizes
post-authentication via Cedar/roles, not OAuth scopes, so there is no scope-insufficiency state at the RS
boundary. **400 malformed** is deliberately folded into the fail-closed **401** path (a malformed/unparseable
bearer token is treated as absent/invalid rather than surfacing token structure to the client). This is an
explicit deferral of the distinct 403/400 status codes, not an oversight; revisit if/when scope-based
authorization is introduced.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MCP client → /mcp/ and /.well-known/oauth-protected-resource | External agent discovers the AS and presents (or lacks) a token |
| operator config → canonical resource URI | Misconfiguration would make clients reject tokens or advertise a wrong AS |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-03-03-01 | Spoofing | canonical resource URI | high | mitigate | resource in metadata MUST equal the identifier clients fetch it by (RFC 9728 §3.3); validated https+no-fragment at startup (fatal); asserted by the metadata test. |
| T-03-03-02 | Denial of Service (lockout) | challenge scope (D-08) | high | mitigate | WWW-Authenticate is emitted ONLY by the /mcp/ wrapper; RequireAuth/ConnectRPC/`/api/*` 401s unchanged — static-credential clients still get a plain 401 and keep working (Pitfall 4). |
| T-03-03-03 | Information Disclosure | metadata endpoint | low | accept | The endpoint is intentionally public (unauthenticated discovery, like /api/auth/oidc/providers); it exposes only issuer URLs already known to clients. |
| T-03-03-04 | Elevation of Privilege | token passthrough | high | mitigate | Confirm the loopback mcpClient hop is internal same-audience to SpecGraph's own ConnectRPC (A3), not an upstream API — no forbidden passthrough. |
| T-03-03-05 | Spoofing/Elevation | AS-advertisement vs binding issuer split (review HIGH #1) | high | mitigate | authorization_servers AND buildClaimsMappingByIssuer both use config.ProviderIssuer(pc), so the advertised AS equals the binding issuer and the claims-mapping key — an MCP client cannot discover a mismatched AS and role mapping cannot silently fail. |
| T-03-03-SC | Tampering | package installs | high | accept | No new packages this phase (RESEARCH Package Legitimacy Audit: none). |
</threat_model>
