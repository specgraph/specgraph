<objective>
Establish the identity-materialization seam that AUTH-01's oauth2 provider and AUTH-04's
token validation both build on, and complete AUTH-05 (session issuer population) on top of it.

Two structural changes, one behavioral change:
1. Change `LoginProvider.Exchange` to return a normalized `*OIDCClaims` (not a raw id_token
   string), and extract the post-verification tail of `resolveJWT` into a shared
   `materializeIdentity(ctx, *OIDCClaims, interactive bool)` helper called by both the
   bearer-JWT path and a new claims-based interactive-login entrypoint `Resolver.ResolveLogin`.
   This is Pattern 2 from RESEARCH.md — it protects D-08 by leaving `Resolve(ctx, token string)`
   structurally untouched.
2. Add `Identity.Issuer` (verified iss for OIDC), thread it through materializeIdentity, and set
   `sess.Issuer` at CreateSession time in handleCallback (AUTH-05, D-09). `CreateSession` already
   persists the issuer column; only the caller currently leaves it empty. No backfill (D-10).
3. Define the phase's full new config surface in one place (global.go) so downstream plans
   (02/03/04) consume config without re-editing the file.

Purpose: A single, low-risk shared seam prevents duplicated provisioning logic and a 1:1
user↔subject assumption (D-03), and makes issuer audit data available for every provider kind.
Output: The seam, the config contract, and AUTH-05 issuer threading — all green under `task test`.

**AUTH-05 CLI-login scope (W2, user-accepted):** Issuer is threaded only through the browser
login callback (`handleCallback` → `CreateSession`). The CLI-broker path (`ExchangeCLICode`,
web_auth.go:230) intentionally leaves `sess.Issuer` empty this phase — GitHub login is not exposed
to `specgraph login` (RESEARCH OQ3), and any OIDC CLI-minted sessions with empty issuer age out
within one session TTL (bounded staleness, consistent with D-10's no-backfill rationale). The
Success Criterion #3 "every web session" wording is therefore satisfied for browser-minted sessions;
the CLI gap is an accepted, bounded deferral, not a silent scope reduction. Threading issuer through
`CreateCLICode`/`ExchangeCLICode` is a fast-follow if CLI login later needs issuer audit.
</objective>
