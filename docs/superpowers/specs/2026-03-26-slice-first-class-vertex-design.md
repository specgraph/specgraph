# Introduce Slice as First-Class Graph Vertex

**Date**: 2026-03-26
**Status**: Approved
**Bead**: spgr-6sw

## Problem

Decomposition slices are stored as Spec nodes distinguished only by COMPOSES edges. This conflates two distinct entities. A Slice has a parent, is auto-generated by decompose, has a simple open/claimed/done lifecycle, and doesn't traverse the authoring funnel. Storing slices as Specs causes dashboard counting confusion, requires edge-based filtering to distinguish them, and makes the data model dishonest.

## Changes

### 1. New Proto: `slice.proto`

Create `proto/specgraph/v1/slice.proto` with:

```protobuf
message Slice {
  string id = 1;
  string slug = 2;             // parent-slug/slice-id (full slug)
  string parent_slug = 3;
  string slice_id = 4;         // local ID within decomposition (e.g. "backend-api")
  string intent = 5;
  repeated string verify = 6;
  repeated string touches = 7;
  repeated string depends_on = 8;  // full sibling slice slugs (resolved at creation)
  SliceStatus status = 9;
  string assigned_to = 10;
  google.protobuf.Timestamp created_at = 11;
  google.protobuf.Timestamp updated_at = 12;
}

enum SliceStatus {
  SLICE_STATUS_UNSPECIFIED = 0;
  SLICE_STATUS_OPEN = 1;
  SLICE_STATUS_CLAIMED = 2;
  SLICE_STATUS_DONE = 3;
}

service SliceService {
  rpc ListSlices(ListSlicesRequest) returns (ListSlicesResponse);
  rpc GetSlice(GetSliceRequest) returns (GetSliceResponse);
  rpc ClaimSlice(ClaimSliceRequest) returns (ClaimSliceResponse);
  rpc CompleteSlice(CompleteSliceRequest) returns (CompleteSliceResponse);
}
```

Request/response messages follow existing patterns (slug-based lookups, standard error codes).

**`depends_on` semantics:** The proto stores full slugs (e.g., `parent/slice-id`), resolved at creation time from the local IDs in the `DecomposeRequest` input. The `DecompositionSlice.depends_on` in the request still uses local IDs -- `StoreDecomposeOutput` resolves them to full slugs.

### 2. Modify DecomposeOutput

In `proto/specgraph/v1/authoring.proto`, slim `DecomposeOutput`. The current `repeated DecompositionSlice slices = 2` is replaced:

```protobuf
message DecomposeOutput {
  DecompositionStrategy strategy = 1;
  reserved 2;
  reserved "slices";
  repeated string slice_slugs = 3;  // references to Slice nodes
}
```

Field 2 is reserved (was `repeated DecompositionSlice slices`). New field 3 holds slug references. `DecompositionSlice` message remains in the file for the `DecomposeRequest` input but is no longer used in output.

**`DecomposeResponse.child_spec_slugs`** (field 6): Reserve this field and name. Add a new `slice_slugs` field to the response:

```protobuf
message DecomposeResponse {
  DecomposeOutput output = 1;
  // ... existing fields ...
  reserved 6;
  reserved "child_spec_slugs";
  repeated string slice_slugs = 7;
}
```

### 3. New Graph Node: `:Slice`

Memgraph `:Slice` node properties:

| Property | Type | Description |
|----------|------|-------------|
| `id` | string | ULID (slc- prefix) |
| `slug` | string | `{parent_slug}/{slice_id}` |
| `parent_slug` | string | Parent spec slug |
| `slice_id` | string | Local ID within decomposition |
| `intent` | string | What this slice accomplishes |
| `verify_json` | string | JSON array of verification criteria |
| `touches_json` | string | JSON array of file paths |
| `status` | string | `open`, `claimed`, `done` |
| `assigned_to` | string | Who claimed it (empty if unclaimed) |
| `created_at` | string | RFC3339 |
| `updated_at` | string | RFC3339 |

Edges:

- `(Slice)-[:BELONGS_TO]->(Project)` -- project scoping (required for all project-scoped queries)
- `(Slice)-[:COMPOSES]->(Spec)` -- slice belongs to parent spec
- `(Slice)-[:DEPENDS_ON]->(Slice)` -- sibling dependency (no `content_hash_at_link` -- slices have no content hash)

### 4. Storage Interface

New `SliceBackend` interface in `internal/storage/`:

```go
type Slice struct {
    ID         string
    Slug       string
    ParentSlug string
    SliceID    string
    Intent     string
    Verify     []string
    Touches    []string
    DependsOn  []string   // full sibling slugs
    Status     SliceStatus
    AssignedTo string
    CreatedAt  time.Time
    UpdatedAt  time.Time
}

type SliceStatus string

const (
    SliceStatusOpen    SliceStatus = "open"
    SliceStatusClaimed SliceStatus = "claimed"
    SliceStatusDone    SliceStatus = "done"
)

type SliceBackend interface {
    CreateSlice(ctx context.Context, slice *Slice) error
    ListSlices(ctx context.Context, parentSlug string) ([]*Slice, error)
    GetSlice(ctx context.Context, slug string) (*Slice, error)
    ClaimSlice(ctx context.Context, slug, assignee string) error
    CompleteSlice(ctx context.Context, slug string) error
}
```

**`ScopedBackend`** (`internal/storage/scoper.go`) must embed `SliceBackend`. The Memgraph `Store` type must implement it.

### 5. StoreDecomposeOutput Changes

`internal/storage/memgraph/authoring.go` StoreDecomposeOutput:

- **Before:** Creates child Spec nodes via `CreateSpec`, then COMPOSES edges between Spec nodes.
- **After:** Creates `:Slice` nodes via `CreateSlice`, then COMPOSES + BELONGS_TO edges. DEPENDS_ON edges between Slice nodes (no `content_hash_at_link`).
- Returns slice slugs for the slimmed `DecomposeOutput`.
- Local IDs in `DecomposeRequest` slices are resolved to full slugs (`parent/id`) during creation.

The domain `DecomposeOutput` struct changes:

```go
type DecomposeOutput struct {
    Strategy   DecompositionStrategy `json:"strategy,omitempty"`
    SliceSlugs []string              `json:"sliceSlugs,omitempty"`
}
```

### 6. Server Handler

New `slice_handler.go` in `internal/server/` implementing `SliceService`:

- `ListSlices` -- returns slices for a parent slug
- `GetSlice` -- returns a single slice by slug
- `ClaimSlice` -- sets status=claimed, assigned_to
- `CompleteSlice` -- sets status=done

Authoring handler's Decompose RPC updated to work with the new Slice storage. `DecomposeResponse` returns `slice_slugs` (new field 7) instead of `child_spec_slugs` (reserved).

### 7. CLI Changes

- `cmd/specgraph/decompose.go` -- update output rendering (shows slice slugs, not inline slice details)
- New `cmd/specgraph/slice.go` -- `slice list <parent-slug>`, `slice get <slug>`, `slice claim <slug>`, `slice complete <slug>`

### 8. Web UI Changes

**Spec detail page** (`web/src/routes/spec/[...slug]/+page.svelte`):

- Decompose accordion changes from reading inline `decomposeOutput.slices` to fetching slices via `sliceClient.listSlices({ parentSlug })`. This is a separate async fetch (like findings) -- non-blocking, with loading/error states.
- Renders slice cards with status badges (open=gray, claimed=blue, done=green)
- Shows `assigned_to` when claimed

**Client** (`web/src/lib/api/client.ts`):

- Add `sliceClient` export for the new `SliceService`

**Graph visualization**:

- `GetFullGraph` returns Slice nodes with `label = "Slice"` (uses existing `GraphNode.label` field -- no new proto field needed, matching the existing `"Spec"` / `"Decision"` pattern)
- Graph renders Slices with a distinct visual (e.g., rounded rectangle vs diamond for Specs)

### 9. GetFullGraph Changes

Use the existing `GraphNode.label` field (currently holds `"Spec"` or `"Decision"` from `labels(n)[0]` in Cypher). The `:Slice` label naturally maps to `"Slice"`. No proto change needed.

The `GetFullGraph` Cypher query in `internal/storage/memgraph/graph.go` must update `WHERE n:Spec OR n:Decision` to `WHERE n:Spec OR n:Decision OR n:Slice`.

Other graph queries that should include Slices:

- `GetDependencies` / `GetTransitiveDeps` -- yes, slices participate in dependency analysis
- `GetImpact` -- yes, slices can be impacted by upstream changes
- `GetReady` -- no, readiness is a Spec concept (slices have their own status)
- `GetCriticalPath` -- yes, slices participate in dependency chains

Add `NodeLabelSlice NodeLabel = "Slice"` to `internal/storage/graph.go`.

## Migration

Clean break. No migration of existing data. Existing decomposed specs in demo projects will lose their child data and need to be re-decomposed.

## Out of Scope

- Slice-level analytical passes (slices are execution units, not authoring units)
- Slice conversation recording (no elicitation happens per-slice)
- Slice content hashing / drift detection
- Nested decomposition (slices of slices)
