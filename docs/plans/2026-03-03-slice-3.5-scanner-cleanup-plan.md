# Slice 3.5 — Scanner Removal & Documentation Cleanup

**Status:** Complete — implemented in PR #22.

> **For Claude:** REQUIRED SUB-SKILL: Use superpowers:executing-plans to implement this plan task-by-task.

**Goal:** Remove the bespoke Go AST codebase scanner and correct all documentation to reflect the intended design: codebase context gathering is agent-driven (LLM reads the code), not a custom static-analysis subsystem.

**Architecture:** The `internal/scanner/` package (Tier 0 file detection, Tier 1/2 Go AST parsing) is replaced by nothing — the design intent is that an LLM agent surveys the codebase during `specgraph init` and authoring, using its built-in code comprehension. The scanner package, the `--scan` CLI flag, and all documentation references to "codebase scanner" as a component are removed or reframed. Site docs also receive quality fixes identified during review (deduplication, status markers, coherence).

**Tech Stack:** Go (code removal), Markdown (doc edits), git

**Sequencing:** Execute after Slice 3 (`feat/slice-3-authoring-funnel`) merges to main. Branch from main.

**Background:** The original design spec (Section 4) describes a three-tier context model (Orientation → Navigation → Deep) for how an agent progressively understands a codebase. This describes agent behavior, not system architecture. The implementation incorrectly built a Go-specific AST parser (`internal/scanner/`) that extracts structural metadata (packages, interfaces, structs, functions) — information that any LLM agent can obtain by reading source files directly. PR #13 (now closed) attempted to document this as an architectural pillar, further overstating its role.
